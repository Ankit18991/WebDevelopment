Constraints In DBMS

Types of Constraints

There Are mainly Four Type of Constraints IN Relational Model:-

1. Domain Constraint
2. Tuple Uniqueness Constraint
3. Key Constraint
4. Entity Integrity Constraint
5. Self Referencial Constraint

Expaination of All the Types :-

1. Domain Constraint

Domain Constraint is a type of Constraint Which states that the Type of Values that can be Inserted in a particular attribute. Attributes can Only have specific values. For example:

1. Age cannot be less than 0.

2. A telephone number can't have a Characters as one of the Digits of the number.

2. Tuple Uniqueness Constraint

It is a contraint in the Relational Model which states that two tuples in a relations can't be the same. It is Because tables are basically relations. And relations (in mathematics) are basicaly Sets(subsets of A X B). And sets can't have duplicate values. For example , the value of all the attribute of two customers can't be the same in the customer table.

3. Key Constraint

It is a contraints in the relation model of DBMS , which states that mimimum number of attributes in the table should UNIQUELY identify a tuple In the Table or relation.

4. Entity Integrity Constraint

It is the Constraint on Primary Key in the table. According to Entity Integrity Constraint, the primary key :

  1. Have to Be UNIQUE
  2. Have to Be NOT NULL
  3. Can Be use to Reference Other table

Importance of Primary Key In RELATIONAL MODEL

Primary key in a table is the most Important aspect to have a relational database where each table is connected to the other. It si basically used for the unique identification of each record. Even if there is no primary key for an entity ,we as a Databse developer ,it is our role to assign the primary key for that entity. We try to Avoid having Weak Enitities.

5. Self - Referencial Constraint

It is the Constraint on Foreign Key. According to this constraint, a Primary Key pf One table or relation can be used as foreign key in other table in order to have a relation between Tables. Following is true According to This constraint:-

  1. CASE 1: If We delete a record from the referenced table:-
    1. Do Nothing
    2. ON DELETE CASCADE
    3. ON DELETE SET NULL
  2. CASE 2: If we delete A record from the referencing table:-
    1. No effect On the referencial Integrity.
  3. CASE 3: If we insert or update a record in the referencing table:-
    1. ON INSERT CASCADE (while Inserting)
    2. NO CHANGE(while updating)
